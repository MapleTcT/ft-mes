# ADP 系统设计逻辑梳理

## 结论

当前交付包不是一套完整 MES 业务产品说明书对应的业务系统，而是一套 ADP/BAP 平台运行包。它的核心定位是“企业信息系统快速开发平台”，用于通过实体配置、流程配置、菜单权限、打印、导入导出、任务调度、门户等平台能力，快速组态或二次开发 MES 生产过程中的业务模块。

因此，当前代码和手册能说明“平台如何搭起来、如何配置业务模块”，但不能完整说明“某个具体 MES 业务产品如何流转”。如果后续收到生产、工单、报工、质检、追溯等业务模块包，需要把它们作为平台上的业务应用接入，并单独补业务验收用例。

## 平台设计逻辑

ADP/BAP 的设计可以理解为四层：

1. 基础运行层
   - Nginx、Gateway、Nacos、Redis、Kafka、Zookeeper、Keycloak、MinIO、数据库。
   - 负责访问入口、服务注册配置、缓存、消息、认证、文件和数据持久化。

2. 平台公共服务层
   - 系统管理、组织人员、权限、菜单、国际化、系统编码、审计日志、电子签名、打印、任务调度、通知、文件服务。
   - 这些服务是业务模块共用的基础能力。

3. 低代码/组态开发层
   - 实体配置是核心：模块、实体、模型、字段、列表视图、编辑视图、查看视图、树视图、参照视图、移动视图、图表控件。
   - 工作流、脚本引擎、Excel 导入导出、打印模板、门户配置等能力用于把实体模型组合成可运行的业务页面。

4. 业务模块层
   - 业务模块应由后续独立包提供，例如生产管理、工单管理、物料/BOM、工艺路线、质量、设备、仓储、追溯等。
   - 当前包只提供平台能力和少量通用基础编码，不包含完整 MES 业务闭环。

## 目前缺失的业务设计资料

如果要做真正 MES 业务测试，当前资料缺少以下内容：

- 业务蓝图：工厂、车间、产线、工位、设备、人员、班组、物料、BOM、工艺路线之间的关系。
- 主数据初始化说明：哪些数据必须先建、字段含义是什么、编码规则是什么。
- 业务流程说明：生产订单到工单、派工、开工、报工、质检、完工入库、追溯的状态机。
- 异常流程说明：报废、返工、暂停、缺料、设备停机、质检不合格。
- 权限矩阵：不同角色能访问哪些菜单和操作。
- 接口说明：业务模块对外 API、事件、消息、第三方系统集成方式。
- 验收用例：每条业务链路的输入、操作步骤、预期数据库和页面结果。

## 后续业务模块接入时的测试思路

后续如果收到业务模块代码包，应按下面顺序处理：

1. 包体识别
   - 后端服务、前端应用、数据库脚本、Nacos 配置、菜单权限、静态资源、依赖组件。

2. 平台接入
   - Docker Compose 增加服务。
   - Nacos 增加配置。
   - Gateway/Nginx 增加路由。
   - 菜单、按钮、权限和角色初始化。
   - 数据库脚本转换到 PostgreSQL。

3. 主数据准备
   - 先构造最小工厂模型：组织、人员、角色、车间、产线、工位、设备、物料、BOM、工艺路线。

4. 最小业务闭环
   - 先跑一条最小链路：生产订单 -> 工单 -> 派工/开工 -> 报工 -> 完工/入库 -> 追溯查询。

5. 扩展验证
   - 质检、异常、设备、库存、报表、权限隔离、日志审计、电子签名、打印和导入导出。

## 对当前测试环境的判断

当前测试环境能验证平台是否可用，包括登录、菜单、权限、基础服务、实体配置、任务调度、打印、主题、系统配置等。但它不能证明完整 MES 业务可用，因为业务模块和业务说明书尚未进入当前包。

后续真正的交付标准应从“页面能打开”提升为“平台可用 + 业务模块闭环可用 + 业务数据结果正确”。
